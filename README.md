# СКУД Analytics — Пилот

Пилотный проект по аналитике нахождения сотрудников на предприятии на основе данных из системы контроля и управления доступом (СКУД Parsec).

> **Важно для агента:** всё выполняется вручную владельцем проекта. Задача агента — разбираться в методологии и писать/исправлять код. Ничего не запускать самостоятельно.

---

## Цель

Рассчитать время нахождения сотрудников в трёх зонах предприятия по данным СКУД.

Целевой формат выгрузки: **сотрудник (id + ФИО) — дата — зона — время пребывания — комментарий**.

Текущий скоуп: данные за **март 2026**, разовый сбор (пилот, без инкрементальной загрузки).

---

## Зоны и их логика

Зоны вложены друг в друга — сотрудник проходит их последовательно:

```
Территория (КПП, общий периметр)
    └── Здание (вход в здание: раздевалки, столовая и т.д.)
            └── Производство (чистые зоны внутри здания)
```

Маппинг конкретных точек прохода (`territory_name` из СКУД) по зонам хранится в `config.py` — для каждой производственной площадки отдельно.

## Методология расчёта

Подробно описана в `methodology.md`. Ключевые моменты ниже.

### Турникеты vs двери

Это принципиально важное различие:

- **Турникет** — через него проходит строго один человек. Событие однозначно соответствует конкретному сотруднику.
- **Дверь** — один человек прикладывает пропуск, но пройти может несколько. Поэтому события на дверях обрабатываются по отдельной логике (см. `methodology.md`).

Тип точки прохода (турникет/дверь) задаётся в `config.py`.

### Ночные смены

Аналитика строится по календарным дням, но сотрудники могут работать в ночную смену — их сессия пересекает полночь. Правила:

- Если последнее событие дня у сотрудника — **вход** (выхода в этом дне нет) → время пребывания засчитывается **до конца дня** (`23:59:59`). В выгрузке проставляется комментарий: `"проставлено до конца дня"`.
- Если первое событие дня у сотрудника — **выход** (входа в этом дне нет) → время пребывания засчитывается **от начала дня** (`00:00:00`).

## Архитектура

```
Два инстанса СКУД (Parsec API)
        ↓
  collect.ipynb
  — запрашивает события по указанным точкам прохода
  — фильтрует: только события входа/выхода сотрудников
  — приводит время к UTC+3
  — пишет сырые данные в PostgreSQL
        ↓
  PostgreSQL
  — parsecnew_employees  (справочник сотрудников)
  — parsecnew_events     (сырые события)
        ↓
  views.sql
  — вся логика расчётов на стороне БД
  — итоговая витрина для выгрузки
```

Вся бизнес-логика (расчёт времени, ночные смены, агрегация по зонам) — **в SQL views**, не в Python. Python только собирает и кладёт сырые данные.

## Структура config.py

Файл не передаётся агенту (содержит credentials). Ниже описание структур, которые в нём лежат.

**Площадки** — список производственных площадок. Каждая площадка содержит:
- название площадки (используется как значение поля `site` в таблице events)
- к какому инстансу СКУД относится (`instance_1` или `instance_2`)

**Инстансы СКУД** — два инстанса Parsec. Каждый содержит:
- API endpoint
- credentials (login/password)

**Маппинг зон** — для каждой площадки: словарь `{territory_name: zone}`, где `zone` — одно из значений `territory` / `building` / `production`. Также для каждой точки указан тип устройства: `turnstile` или `door`.

**Подключение к БД** — строка подключения к PostgreSQL.

> **[ ДОЗАПОЛНИТЬ ]** Уточнить фактические названия ключей/переменных в config.py, если агенту нужно будет на них ссылаться в коде.

## База данных

Все таблицы имеют префикс `parsecnew_`. DDL в `sql/ddl.sql`.

### `parsecnew_employees`

| Поле | Описание |
|---|---|
| `id` | ID сотрудника в СКУД |
| `full_name` | ФИО |
| `department` | Департамент / подразделение |

### `parsecnew_events`

| Поле | Описание |
|---|---|
| `employee_id` | ID сотрудника |
| `event_time` | Время события (UTC+3) |
| `direction` | `enter` / `exit` |
| `point_name` | Название точки прохода |
| `zone` | `territory` / `building` / `production` |
| `site` | Название площадки |

> **[ ДОЗАПОЛНИТЬ ]** Уточнить итоговый список полей по актуальному `ddl.sql` — выше примерная схема.
