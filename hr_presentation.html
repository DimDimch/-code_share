01 / 06 Состояние

Где мы сейчас

Честная картина — что работает, что нет, где разрывы

✅ Что уже есть

—HR-метрики для руководителей (численность, зарплата, текучесть)
—Дашборд качества данных «Потока» (рекрутмент)
—СКУД-аналитика присутствия на территории
—Локальные LLM по API — инфраструктура есть

⚠️ Что не работает

—Метрики без контекста — нет бенчмарков, нет «что делать»
—Performance score — в Excel, не в SQL
—Нет связи между доменами: найм ↔ текучесть ↔ performance
—Исторические данные грязные — обучение моделей пока не вариант
💡

Данные есть — но они не превращаются в действия. Руководитель смотрит на цифры и не знает, хорошо это или плохо, и что делать дальше.

03 / 06 СКУД

Аналитика присутствия

СКУД → структурированная отчётность по нахождению сотрудников на предприятии
🏭
Что уже построено
Методология расчёта присутствия по вложенным зонам (территория → здание → производственный участок). Правила инференса пропущенных событий, уровни доверия к данным.
Методология готова
📊
Что можно добавить
Отчёт для HR — присутствие в разрезе подразделений, отклонения от графика, аномалии (группы прибытия, пропуски). Сигналы для HR BP: сотрудник отсутствовал N дней подряд без отметки в системе.
В разработке
🔗
Связь с HR-данными
СКУД × кадровая система — сопоставить присутствие с подразделением, графиком работы, статусом сотрудника. Потенциал: ранние сигналы риска оттока через изменение паттернов присутствия.
Идея для обсуждения

05 / 06 Self-Service

Data Self-Service через OpenMetadata

Каталог данных как фундамент — разработчики и HR-команда сами генерят сервисы через Cursor
📚
OpenMetadata — каталог всех данных и метрик
Все таблицы, поля, логика расчёта метрик описаны в одном месте. Связи между источниками видны явно. Разработчик понимает что есть и как это считается — без звонка аналитику.
Инфраструктура есть
🤖
Cursor + Skill-файлы = разработчик без аналитика
Мы готовим skill-файлы: описания схем, правила расчёта, примеры запросов. Разработчик или HR-сотрудник подключает их в Cursor → может сам написать сервис на реальных данных, не изобретая логику с нуля.
Идея для совместного пилота
🎯
Что это даёт вам
Ваша команда строит AI-сервисы (tone-of-voice чекер и т.д.) — теперь у неё есть доступ к структурированным HR-данным через описанные скиллы. Быстрее, надёжнее, без разрывов логики.
06 / 06 Процесс

Формат взаимодействия

Как синхронизироваться — без лишних встреч, с понятными приоритетами
Рекомендую
📋
Доска в Jira
Отдельный проект / доска для HR-аналитики. Эпики по доменам: Рекрутмент, Удержание, СКУД, GenBI. Задачи с приоритетами — видно что в работе, что ждёт. Предложить встречу для согласования структуры.
Автоматизируем
🤖
Спринт-саммари через LLM
Раз в 2 недели LLM читает закрытые задачи из Jira и генерит краткий отчёт: что сделано, что задержалось, на что обратить внимание. Вам — 1 мин на прочтение вместо созвона.
📅
Регулярная синхронизация
30 мин раз в 2 недели — только стратегия и разблокировка. Операционка — в Jira. Предлагаю: поставить первую встречу для выработки приоритетов на квартал.
💬
Чат / канал
Отдельный чат для оперативных вопросов. Можно туда же присылать автоматические алерты — упала метрика, сработал сигнал риска, готов новый отчёт.
🚀

Следующий шаг прямо сейчас: договориться о структуре Jira-доски и первом спринте. Я предлагаю сфокусироваться на бенчмарках в HR-метриках + переносе performance score в SQL.
